# Skill 包构建流程（5 步详细指南）

> 本文档是按当前简化协议从零构建 Skill 包的完整操作手册。
> 创建规范以同目录编号文档为准：核心总则见 `001_XGJK_SKILL_PROTOCOL.md`，详细规则见 `002`-`008`。

## Step 1：理解业务与确认范围

先把目标说清楚，再开始建包。

至少确认以下问题：

1. 这个 Skill 解决什么问题？
2. 谁会触发它？用户会怎么说？
3. 需要拆成几个模块？
4. 每个模块下有哪些明确动作？
5. 哪些动作是读操作，哪些动作是写操作？
6. 每个动作需要哪种鉴权模式？

这一阶段的交付物：

- Skill 名称
- Skill 描述
- `skillcode`
- 模块清单
- 动作清单
- 各动作的鉴权模式

## Step 2：创建目录骨架

按最小结构先搭目录：

```bash
mkdir -p <skill-name>/references/<module>
mkdir -p <skill-name>/scripts/<module>
```

然后立即确认以下规则：

- 目标 Skill 统一依赖 `cms-auth-skills`
- 目标 Skill 不实现本地登录流程
- 运行时日志与状态写到工作区根目录 `.cms-log/`

这一阶段只做结构，不写实现细节。

## Step 3：先写 SKILL.md

`SKILL.md` 是总索引，应该先确定下来。

必须完成：

1. YAML 基本字段
2. 能力概览
3. 统一规范
4. 授权依赖说明
5. 输入完整性规则
6. 建议工作流
7. 脚本使用规则
8. 路由与加载规则
9. 宪章
10. 模块路由表
11. 能力树

这一阶段的目标不是“把内容写长”，而是把边界、入口和路由写准。

## Step 4：按模块逐个补全说明与脚本

对每个模块按以下顺序处理：

1. 编写 `references/<module>/README.md`
2. 编写 `scripts/<module>/<action>.py`
3. 更新 `scripts/<module>/README.md`
4. 回填 `SKILL.md` 中对应的模块路由表

### 4.1 模块说明至少写清楚

- 模块是干什么的
- 用户会在什么场景下触发
- 需要哪些输入
- 每个动作做什么
- 每个动作使用什么鉴权模式
- 输出结果如何理解

### 4.2 Python 脚本至少做到

- 可以直接命令行运行
- 参数名与文档一致
- 鉴权读取方式明确
- 请求超时明确
- 重试策略明确
- 输出结构明确

### 4.3 每完成一个动作就做一次迷你自检

- 路由表里的脚本路径是否存在
- 模块说明里的动作名是否覆盖该脚本
- 脚本参数与说明是否一致
- 鉴权模式在说明、脚本索引、脚本实现中是否一致

## Step 5：按清单完成三轮复核并交付

### 第一轮：逐项打勾

按 `008_XGJK_SKILL_VALIDATION_CHECKLIST.md` 逐条检查，不能跳项。

### 第二轮：交叉一致性检查

至少核对以下内容：

- `SKILL.md` 路由表和实际脚本是否一致
- 模块说明和脚本索引是否一致
- 实际目录和能力树是否一致
- 文档中的动作名、模块名、路径是否一致

### 第三轮：手工 smoke test

至少挑选一个读操作和一个写操作脚本做人工验证：

- 读操作验证输出格式
- 写操作验证确认流程和错误处理

全部通过后，再输出最终结论：

```text
✅ Skill 构建完成
- Skill 名称：<name>
- 模块数量：M
- 脚本数量：N
- 清单检查：全部通过
- 手工 smoke test：已完成
- 遗留问题：无
```

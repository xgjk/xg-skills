# Spec-Lite: RT-002 - ARC Reactor 完整设计方案

## 1. ARC 是什么

**ARC = Acquire / Research / Catalogue**

一个面向个人知识建设的 AI Skill。用户提供一个链接、一段描述、或一个研究课题，ARC 自动完成"获取→研究→编目"的全链路，最终输出一份结构化的调研报告并归入个人知识库。

---

## 2. 输入识别与分流路由

### 2.1 输入类型分类

ARC 在接收到用户输入后，首先执行**输入分类**，判断属于以下哪种形态：

| 类型代号 | 输入特征 | 示例 |
|----------|----------|------|
| `URL_ARTICLE` | HTTP(S) 链接，指向文章/博客 | 头条、Medium、博客链接 |
| `URL_VIDEO` | YouTube / B 站等视频链接 | `youtu.be/xxx`, `bilibili.com/video/xxx` |
| `URL_REPO` | GitHub / GitLab 仓库链接 | `github.com/org/repo` |
| `URL_PAPER` | 指向 PDF / 论文的链接 | arxiv, Google Scholar |
| `URL_SOCIAL` | 社交媒体帖子（X/Reddit/Discord） | `x.com/user/status/xxx` |
| `TOPIC` | 纯文本描述，无 URL | "帮我调研一下 DeerFlow 2.0" |
| `FILE` | 用户直接发送的文件/截图 | 粘贴的图片或文档 |

### 2.2 路由决策树

```
用户输入
  ↓
是否包含 URL？
  ├─ 是 → URL 类型识别
  │     ├─ 视频平台 → VIDEO 管道（转录优先）
  │     ├─ GitHub  → REPO 管道（README/架构分析）
  │     ├─ PDF/论文 → PAPER 管道（文本提取+摘要）
  │     ├─ 社交媒体 → SOCIAL 管道（帖子+评论提取）
  │     └─ 其他网页 → ARTICLE 管道（正文提取）
  └─ 否 → 文本/文件
        ├─ 含文件附件 → FILE 管道（OCR/文本提取）
        └─ 纯文本 → TOPIC 管道（多源搜索聚合）
```

## 3. 去重检测与内容合并（Dedup & Merge）

> **核心原则：同一知识主体只维护一份"活报告"，多源信息增量合并，而非重复堆叠。**

### 3.1 去重检测流程

在 ARC 进入 Acquire 阶段**之前**，必须先执行去重检测：

```
用户输入
  ↓
路由分类（URL_ARTICLE / URL_VIDEO / TOPIC 等）
  ↓
去重检测 ←── 扫描已有知识库
  ├─ 未命中 → 创建新调研（正常 ARC 流程）
  ├─ 命中同一主体 → 进入合并模式（Merge Mode）
  └─ 疑似相关 → 提示用户确认是否合并
```

### 3.2 检测维度（三级匹配）

| 级别 | 检测方式 | 示例 | 结果 |
|------|----------|------|------|
| **L1: URL 精确匹配** | 完全相同的 URL | 用户发了同一个头条链接 | 直接跳过，提示"已调研" |
| **L2: 主体匹配** | 从内容中提取的核心实体名 | 头条讲 DeerFlow + 抖音讲 DeerFlow | 触发合并模式 |
| **L3: 语义相似** | 标题/摘要语义相似度 | "字节超级智能体"与"DeerFlow 2.0 拆解" | 提示用户确认 |

### 3.3 主体识别规则

每次调研完成后，在 `knowledge/entities/` 中为核心主体建立实体页。实体页的唯一标识 = **规范化主体名**（如 `DeerFlow`、`Karpathy`、`Caveman`）。

后续新输入进来时：
1. 提取输入中的关键实体名
2. 与 `knowledge/entities/` 目录做模糊匹配
3. 命中 → 判断为同一主体

### 3.4 合并模式（Merge Mode）

当检测到新输入与已有报告属于**同一主体**时，ARC 不再新建报告，而是执行合并：

#### Step 1: 增量信息提取
- 仅提取新源中**旧报告不包含的信息**
- 标注每条新增信息的来源（`[来源: 抖音/白白说大模型]`）

#### Step 2: 报告追加更新
在已有报告中：
- **基本信息表**：追加新的来源行
- **正文章节**：新增发现插入对应章节末尾，或新增独立章节
- **附录**：追加新的原始素材路径

#### Step 3: 源追踪表
每份报告底部维护一张**多源追踪表**：

```markdown
## 源追踪

| # | 来源平台 | URL | 内容类型 | 采集日期 | 贡献内容 |
|---|----------|-----|----------|----------|----------|
| 1 | 今日头条 | https://... | 文章 | 2026-04-06 | 初始调研 |
| 2 | 抖音 | https://... | 视频 | 2026-04-07 | 补充架构细节 |
| 3 | YouTube | https://... | 视频 | 2026-04-08 | 新增性能对比数据 |
```

#### Step 4: 实体页同步更新
合并后自动更新 `knowledge/entities/[主体].md` 的：
- 信息来源列表
- 最后更新时间
- 知识完整度评估

### 3.5 全自动执行规则

> **核心理念：ARC 必须自主决策，绝不因"拿不准"就打断用户。**

| 场景 | ARC 行为 |
|------|----------|
| L1 精确重复 | 直接跳过，汇报"此链接已于 X 日调研过"并给出报告链接 |
| L2 同主体新源 | **静默进入合并模式**，完成后汇报"已将新信息合并至 [报告名]" |
| L3 疑似相关 | **自主判断**：若核心主体 ≥70% 重叠 → 合并；若仅部分关联 → 独立建档但在两份报告中互相标注关联；若一个输入涉及多个主体 → **一分为二**，各自归入对应实体 |
| 用户明确要求独立 | 尊重用户指令覆盖自动判断 |

### 3.6 冲突处理

当新源信息与旧报告数据矛盾时（如 Star 数变化、结论相反）：
- **数据类矛盾**（Star 数、版本号）：直接**用新值覆盖旧值**，并标注更新时间
- **观点类矛盾**（评价、结论）：**两者并存**，标注各自来源，交由用户裁定

## 4. A 阶段 —— Acquire（获取）

### 4.1 各管道的获取方法

| 管道 | 获取方式 | 回退方案 |
|------|----------|----------|
| ARTICLE | 浏览器抓取网页正文 → 提取主体内容 | 搜索引擎缓存 |
| VIDEO | 字幕/转录提取（Whisper/平台 API） | 视频简介 + 评论区摘要 |
| REPO | GitHub API → README + 目录结构 + Star/Issue 统计 | 克隆仓库本地分析 |
| PAPER | PDF 文本提取 → 结构化解析 | 搜索论文摘要 |
| SOCIAL | 帖子正文 + 核心评论提取 | 截图 OCR |
| TOPIC | 多搜索引擎聚合（≥3 个源） | 逐一搜索并合并 |
| FILE | OCR / 文本提取 | 人工辅助 |

### 4.2 获取质量门控

每次获取完成后，执行以下检查：
- [ ] **内容非空检查**：提取到的有效文本 ≥ 200 字（否则标记获取失败）
- [ ] **语言识别**：检测并记录原文语言
- [ ] **源可信度标记**：标注来源类型（官方文档 / 自媒体 / 论坛 / 论文）

## 5. R 阶段 —— Research（研究）

### 5.1 结构化分析

对获取到的原始内容执行：

1. **核心信息提取**：项目/产品是什么、解决什么问题、技术方案、关键数据
2. **分章节组织**：按逻辑主题切分为 3-8 个独立章节
3. **关键数据表格化**：Star 数、融资额、性能指标等用表格呈现

### 5.2 🔍 二次复检与论证（Verification Pipeline）

> **核心原则：不盲信任何单一信源，对关键声明必须交叉验证。**

#### Step 1: 声明分类
将内容中的关键声明分为三类：
- **客观事实**（Star 数、融资额、发布日期）→ 可直接验证
- **性能/效果声明**（"省 93% Token"、"提速 10x"）→ 需要条件限定
- **主观评价**（"最好的"、"颠覆性的"）→ 标记为观点

#### Step 2: 交叉验证
对核心客观声明执行：
- **数据验证**：直接访问 GitHub/官网确认 Star 数、版本号等
- **条件限定**：性能声明必须标注测试条件（"在 X 场景下"）
- **多源对比**：同一信息至少在 2 个独立源中出现才标记为"已验证"

#### Step 3: 可信度标注
每份报告必须包含可信度评级：

| 等级 | 标识 | 含义 |
|------|------|------|
| ✅ 已验证 | `[VERIFIED]` | 多源确认，数据可靠 |
| ⚠️ 待验证 | `[UNVERIFIED]` | 仅单源，暂无交叉验证 |
| ❌ 存疑 | `[DISPUTED]` | 发现矛盾信息或过度夸大 |
| 💬 观点 | `[OPINION]` | 主观评价，非事实陈述 |

#### Step 4: 风险矩阵（可选）
对于重要调研，生成一份简单的风险/真实性矩阵表。

## 6. C 阶段 —— Catalogue（编目）

### 6.1 标准报告模板

所有 ARC 输出必须遵循以下统一模板：

```markdown
# [主题] 调研报告

> 每份报告自包含完整，开箱即用。

---

## 基本信息

| 字段 | 内容 |
|------|------|
| **主题** | [标题] |
| **来源** | [来源平台] |
| **原文 URL** | [链接] |
| **内容类型** | [ARTICLE/VIDEO/REPO/PAPER/SOCIAL/TOPIC] |
| **调研日期** | [YYYY-MM-DD] |
| **调研人** | [Agent 名称] |
| **可信度** | [VERIFIED / UNVERIFIED / DISPUTED] |

---

## 一、[第一章标题]
...

## 二、[第二章标题]
...

---

## 结论与建议
- 关键发现摘要
- 对我们的价值判断
- 建议后续行动

---

## 附录
- **关联调研**：[相关 RT 或报告链接]
- **原始素材**：[转录稿/截图等存储路径]
- **验证记录**：[哪些声明被验证/存疑]
```

### 6.2 本地存储架构

```
workspace/
├── reports/                     # 报告主目录
│   ├── YYYY-MM-DD/              # 按日期归档
│   │   ├── [主题]-调研报告.md    # 最终报告
│   │   └── raw/                 # 原始素材（转录、截图等）
│   └── index.md                 # 报告索引（自动维护）
```

**命名规范**：`[主题简称]-调研报告.md`（全中文，便于 Obsidian 检索）

### 6.3 分类标签体系

每份报告在 index.md 中标注分类标签：

| 标签 | 说明 |
|------|------|
| `#工具` | 开发工具、SDK、框架 |
| `#架构` | 系统设计、架构模式 |
| `#AI` | AI/LLM 相关技术 |
| `#产品` | 产品分析、竞品调研 |
| `#方法论` | 工作方法、流程设计 |
| `#人物` | 人物/团队调研 |

## 7. 导出调度接口（Dispatcher）

### 7.1 架构设计

ARC 的存储层采用**插件式调度器**架构，解耦"生成报告"和"存到哪"：

```
ARC 核心引擎
  ↓ 生成报告
Dispatcher（调度器）
  ├── local-workspace  （默认，写入 workspace/reports/）
  ├── obsidian-sync     （同步到 Obsidian 文件夹）
  ├── notion-export     （未来扩展）
  └── custom-webhook    （未来扩展，推送到任意目标）
```

### 7.2 Obsidian 同步规则（首期实现）

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `obsidian_vault_path` | `~/Library/Mobile Documents/iCloud~md~obsidian/Documents/` | Obsidian 仓库根目录 |
| `obsidian_target_dir` | `github分享/AI调研/{date}/` | 按日期归档目录 |
| `auto_sync` | `true` | 报告完成后是否自动同步 |

### 7.3 扩展接口定义

未来新增存储目标只需实现一个简单的接口：
- `dispatch(report_path, config)` → 将报告文件发送到目标位置
- 配置项定义在 `references/dispatchers/` 目录下

## 8. 完整流转示例

```
用户: "帮我调研一下这个 https://github.com/bytedance/DeerFlow"

ARC 启动:
  [A] 识别为 URL_REPO → REPO 管道
  [A] 获取 README + 目录结构 + Star/Issue 统计
  [A] 质量门控: ✅ 内容 2800 字，语言中英混合，源=官方仓库

  [R] 结构化分析: 5 个章节（概述/架构/特性/对比/结论）
  [R] 二次复检: Star 数 ✅ 已验证，"超越 AutoGPT" ⚠️ UNVERIFIED
  [R] 可信度: VERIFIED (核心事实已交叉验证)

  [C] 生成报告: DeerFlow-调研报告.md
  [C] 本地存储: workspace/reports/2026-04-08/DeerFlow-调研报告.md
  [C] Obsidian 同步: AI调研/2026-04-08/DeerFlow-调研报告.md
  [C] 更新 index.md: +1 条记录，标签 #工具 #AI #架构
  [C] git commit
```

## 9. 借鉴 Karpathy LLM Wiki 的知识进化机制

> 参考文档：Karpathy LLM Wiki 概念解析 & 进展报告（2026-04-08）

### 9.1 核心启发：从"存储信息"到"演化知识"

Karpathy 的 LLM Wiki 提出了一个颠覆性理念：**不做 RAG，让 LLM 直接维护一套 Markdown Wiki**。这与 ARC 的目标高度一致——我们不只是"调研完扔一份报告"，而是要让每次调研都成为个人知识库的**增量编译**。

### 9.2 引入三层架构到 ARC

借鉴 LLM Wiki 的三层设计，ARC 的 Catalogue 阶段将升级为：

```
┌──────────────────────────────────────────────────┐
│              ARC 知识三层架构                      │
├──────────────────────────────────────────────────┤
│                                                    │
│  Layer 1: Raw（原始素材层）                        │
│  ┌────────────────────────────────┐               │
│  │ reports/YYYY-MM-DD/raw/         │               │
│  │ 转录稿、截图、PDF、网页快照    │               │
│  │ 只读存储，不做筛选，先全收     │               │
│  └────────────────────────────────┘               │
│           ↓                                        │
│  Layer 2: Wiki（知识编译层）                       │
│  ┌────────────────────────────────┐               │
│  │ knowledge/                      │               │
│  │ ├── entities/   (实体页)        │               │
│  │ ├── concepts/   (概念页)        │               │
│  │ ├── comparisons/(对比页)        │               │
│  │ └── index.md    (知识导航)      │               │
│  └────────────────────────────────┘               │
│           ↓                                        │
│  Layer 3: Schema（规则层）                         │
│  ┌────────────────────────────────┐               │
│  │ references/knowledge-rules.md   │               │
│  │ 交叉引用规则、矛盾修正规则    │               │
│  └────────────────────────────────┘               │
│                                                    │
└──────────────────────────────────────────────────┘
```

### 9.3 知识编译（Knowledge Compilation）

每次 ARC 完成一份调研报告后，额外执行一步**知识编译**：

1. **实体提取**：从报告中识别关键实体（项目名、人物、公司）
   - 如果 `knowledge/entities/DeerFlow.md` 已存在 → **追加更新**
   - 如果不存在 → **新建实体页**
2. **概念关联**：识别报告中的核心概念，更新 `knowledge/concepts/`
3. **交叉引用**：在相关实体页之间建立 `[[双向链接]]`（Obsidian 原生格式）
4. **索引更新**：更新 `knowledge/index.md` 目录

**关键原则**：报告是"一次性快照"，Wiki 是"持续演化体"——每次新调研都在**增量编译**你的个人知识图谱。

### 9.4 "质疑自我假设" 机制（Self-Challenge）

借鉴 Claudeopedia 的 Cron 自检模式：

- **触发时机**：当 ARC 发现新调研的内容与已有知识页存在矛盾时
- **行为**：自动生成一份 `[CONFLICT]` 标记的对比笔记，列出：
  - 旧知识页的观点
  - 新调研的矛盾点  
  - 建议修正方案
- **产出**：写入 `knowledge/conflicts/YYYY-MM-DD-[主题].md`
- **用户确认**：矛盾修正必须经过用户确认后才写入实体页

| 阶段 | 内容 | 优先级 |

| **v1.1** | 二次复检管线 + 可信度标注 | P0 |
